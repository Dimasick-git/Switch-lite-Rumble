# Техническая спецификация: Вибро-мод для Switch Lite через игровой слот

Этот документ описывает техническую реализацию модуля тактильной отдачи (Rumble),
подключаемого через слот игрового картриджа — это целевой путь проекта. Основное
внимание уделено питанию из слота и передаче данных.

> **Статус:** это «слотовый» путь — основная цель (см. [DESIGN-NOTES.md](./DESIGN-NOTES.md)
> §0, §12). Часть про Lotus3 и питание слота Dimasick ведёт сам; этот документ —
> рабочая спецификация. Внешний USB-C/BT остаётся запасным путём
> ([concept-external-approach.md](./concept-external-approach.md)).

---

## 1. Архитектура системы перехвата (Software)

Для получения данных о вибрации необходимо вмешательство в работу системной службы **HID (Human Interface Device)**. В консолях Switch Lite команды вибрации генерируются играми, но игнорируются драйвером встроенного контроллера.

### Точка перехвата: IPC MITM
Необходимо реализовать системный модуль (sysmodule) на базе Atmosphere, который будет выполнять роль посредника (MITM) для сервиса `hid`.

| Сервис | Команда | ID | Описание |
| :--- | :--- | :--- | :--- |
| `hid` | `SendVibrationValue` | 201 | Отправка одного значения вибрации |
| `hid` | `SendVibrationValues` | 206 | Пакетная отправка значений для нескольких моторов |

> Примечание: в первой редакции документа ID были указаны как 153/154 — это ошибка.
> Корректные значения (по switchbrew/libnx): 201 и 206. Подробнее — в
> [`DESIGN-NOTES.md`](./DESIGN-NOTES.md) §8a.

**Механика работы:**
1. Сисмодуль регистрирует MITM-обработчик для порта `hid`.
2. При вызове `SendVibrationValue` модулем игры, сисмодуль перехватывает структуру `VibrationValue` (амплитуда и частота для низких и высоких частот).
3. Данные упаковываются в компактный протокол и передаются на конечное устройство.

## 2. Обход защиты игрового слота (The Bypass)

Слот картриджа управляется специализированным чипом **Lotus3 ASIC**. Это главный барьер, который блокирует питание и данные для неавторизованных устройств.

### Механизм блокировки Lotus3
При вставке устройства (замыкание контакта **CD#** на землю):
1. Система загружает прошивку в Lotus3.
2. Подается инициализирующее питание (**VCC 1.8V** и **3.1V**).
3. Lotus3 пытается прочитать **Card Header** и выполнить процедуру **Challenge-Response** (аутентификация).
4. Если аутентификация провалена, Lotus3 отключает линии питания и переводит шину в состояние ошибки.

### Методы обхода для получения питания и данных

#### Вариант А: Эмуляция протокола (Hardware Emulation)
Создание устройства на базе быстрого микроконтроллера или FPGA, способного отвечать на команды Lotus3.
- **Сложность:** Экстремальная. Протокол использует CRC-32 и защищенный режим (AES-128-CTR).
- **Плюс:** Полная автономность, не требует глубоких патчей системы.

#### Вариант Б: Программный патч FS-сервиса (Software Patching)
Модификация службы `fs` (Filesystem), которая управляет `IDeviceOperator`.
- **Суть:** Патчинг проверок в функции `nn::fs::detail::IDeviceOperator::GetGameCardHandle`.
- **Задача:** Заставить систему игнорировать статус ошибки Lotus3 и не подавать команду на отключение питания (**Power Gating**).
- **Риск:** Lotus3 может иметь аппаратный тайм-аут на подачу питания без успешной инициализации.

## 3. Электропитание и ограничения

Слот картриджа имеет жесткие ограничения по току, не рассчитанные на индуктивные нагрузки (моторы).

| Параметр | Значение | Примечание |
| :--- | :--- | :--- |
| **VCC 3.1V** | ~50-100 мА | Основное питание чипа картриджа |
| **VCC 1.8V** | ~20-50 мА | Питание I/O логики |
| **Пик Taptic Engine** | >300 мА | Требуется буферный конденсатор или АКБ |

**Рекомендация:** Для предотвращения перезагрузок консоли из-за просадок напряжения, устройство в картридже должно иметь **ионистор (supercapacitor)** или миниатюрный Li-Po аккумулятор, который подзаряжается от слота в моменты покоя.

## 4. Канал передачи данных (Rumble Data)

Хотя питание берется из слота, передача данных о вибрации через линии **DAT0-DAT7** требует реализации проприетарного протокола.

1. **Путь через шину картриджа:** Сисмодуль должен отправлять данные в `fs`, который перешлет их в Lotus3, а тот — на линии DAT. Это требует написания кастомного драйвера для Lotus3.
2. **Альтернативный путь:** Использование линий данных слота как обычного UART/SPI (возможно только при полной деактивации штатного драйвера Gamecard в ядре/FS).

---

## 5. Вывод о реализуемости

Реализация "через слот" с обходом аутентификации — это проект уровня **Hardcore Reverse Engineering**. 

1. **Питание:** Достижимо через программные патчи сервиса `fs`, заставляющие систему держать питание включенным несмотря на ошибки.
2. **Данные:** Проще всего использовать USB-C или Bluetooth, но если цель — именно слот, потребуется эмуляция базового протокола картриджа.
3. **Безопасность:** Существует риск повреждения контроллера питания Lotus3 при превышении лимитов по току вибромотором.

**Вердикт:** Проект технически возможен при условии использования кастомного патча для `fs-srv` и аппаратного буфера питания на стороне картриджа.
