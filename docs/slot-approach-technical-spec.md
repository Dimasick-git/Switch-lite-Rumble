# Technical spec: Switch Lite rumble mod through the game card slot

This document describes the technical implementation of a tactile-feedback (Rumble)
module connected through the game card slot — the project's primary path. The focus is
slot power and data transfer.

> **Status:** this is the "slot" path — the primary goal (see
> [DESIGN-NOTES.md](./DESIGN-NOTES.md) §0, §12). The Lotus3 and slot-power part is led
> within the project; this document is the working spec. External USB-C/BT remains the
> fallback ([concept-external-approach.md](./concept-external-approach.md)).

---

## 1. Interception architecture (software)

To obtain vibration data, the **HID (Human Interface Device)** system service must be
intercepted. On the Switch Lite, vibration commands are generated by games but ignored
by the built-in controller's driver.

### Interception point: IPC MITM
An Atmosphere-based system module (sysmodule) acts as a man-in-the-middle (MITM) for
the `hid` service.

| Service | Command | ID | Description |
| :--- | :--- | :--- | :--- |
| `hid` | `SendVibrationValue` | 201 | Send a single vibration value |
| `hid` | `SendVibrationValues` | 206 | Batch-send values for several motors |

> Note: an early draft listed these IDs as 153/154 — that was wrong. The correct
> values (switchbrew/libnx): 201 and 206. More in
> [`DESIGN-NOTES.md`](./DESIGN-NOTES.md) §8a.

**Mechanism:**
1. The sysmodule registers a MITM handler for the `hid` port.
2. When a game calls `SendVibrationValue`, the sysmodule intercepts the
   `VibrationValue` structure (amplitude and frequency for the low and high bands).
3. The data is packed into a compact protocol and sent to the end device.

## 2. Getting power and data past the slot protection

The cartridge slot is managed by the dedicated **Lotus3 ASIC** — the main barrier that
blocks power and data for unauthorized devices.

### Lotus3 gating mechanism
On insertion (the **CD#** contact shorted to ground):
1. The system loads firmware into Lotus3.
2. Init power is applied (**VCC 1.8V** and **3.1V**).
3. Lotus3 reads the **Card Header** and runs the **Challenge-Response**
   (authentication) procedure.
4. On authentication failure, Lotus3 cuts the power lines and puts the bus into an
   error state.

### Approaches to obtain power and data

#### Option A: Protocol emulation (hardware emulation)
A device on a fast microcontroller or FPGA able to answer Lotus3's commands.
- **Difficulty:** extreme. The protocol uses CRC-32 and a secure mode (AES-128-CTR).
- **Plus:** fully autonomous, no deep system patches.

#### Option B: `fs` service software patch
Modifying the `fs` (Filesystem) service that drives `IDeviceOperator`.
- **Idea:** patch the checks in `nn::fs::detail::IDeviceOperator::GetGameCardHandle`.
- **Goal:** make the system ignore the Lotus3 error state and not issue the **power-
  gating** command.
- **Risk:** Lotus3 may have a hardware timeout on power without successful init.

## 3. Power and limits

The slot has tight current limits, not designed for inductive loads (motors).

| Parameter | Value | Note |
| :--- | :--- | :--- |
| **VCC 3.1V** | ~50–100 mA | Main power for the cartridge chip |
| **VCC 1.8V** | ~20–50 mA | I/O logic power |
| **Taptic Engine peak** | >300 mA | Needs a buffer capacitor or battery |

**Recommendation:** to prevent console resets from voltage sag, the cartridge device
should carry a **supercapacitor** or a tiny Li-Po that charges from the slot while
idle.

---

## 4. Data channel (Rumble Data)

Even with power from the slot, sending vibration data over the **DAT0-DAT7** lines
requires implementing a proprietary protocol.

1. **Through the cartridge bus:** the sysmodule sends data to `fs`, which forwards it
   to Lotus3, which drives the DAT lines. This requires a custom Lotus3 driver.
2. **Alternative:** use the slot data lines as plain UART/SPI (only possible if the
   stock Gamecard driver in the kernel/FS is fully deactivated).

---

## 5. Feasibility

Implementing "through the slot" with an authentication bypass is a **hardcore
reverse-engineering** level project.

1. **Power:** achievable via `fs` software patches that keep power on despite errors.
2. **Data:** easiest over USB-C or Bluetooth, but if the slot is the target, basic
   cartridge-protocol emulation is required.
3. **Safety:** there's a risk of damaging the Lotus3 power controller if the vibration
   motor exceeds current limits.

**Verdict:** technically possible given a custom `fs-srv` patch and a hardware power
buffer on the cartridge side.
